git rebase Considered Harmful

What is git rebase?

  • Dangerous
  • A Terrible Hack
  • A Source of Bugs
  • A Lie

Lots of people like it because they miss the merge conflicts linear history that legacy version control systems had.

More specifically it is a dangerous terrible hack that lets you lie about your commit history to make it appear that you all took turns making commits one-at-a-time.

Dangerous

git rebase is dangerous because it destroys commit history.

It takes commits off of your branch and "replays" them on top of the branch you want to rebase onto. These "replayed" commits have no relation to your previous commits.

It's like you've kidnapped the originals and replaced them with clones.

And like clones they don't have the same parents as the originals.

A Terrible Hack

git rebase is a terrible hack because it doesn't preserve relationships.

In Git all of the logic that it uses to tell what changes are made between different commits and how to resolve conflicts is dependent on parent/child relationships.

This is especially true in the case where you're merging multiple branches into a single one over time.

Without this information you often wind up resolving the same conflicts multiple times or sometimes even dropping commits.

A Source of Bugs

git rebase is a source of bugs because it causes more merge conflits.

Having additional conflicts means that you wind up resolving more merge conflicts. Since manually resolving merge conflicts is an error prone process every time resolve one you increase your odds of introducing a bug.

This is especially true because you pay a lot of attention the first time you resolve a merge conflict but after that it's human nature to skim over it.

A Lie

git rebase is a lie because you didn't actually take turns writing commits one at a time!

This is a somewhat minor issue compared to the above but not having the branching history accurately reflected can make it much harder to tell when a feature was started and finished.

Please be honest with Git and with yourself, unless it's your team's policy don't rebase your code.
